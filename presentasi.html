Menjelaskan 8 tabel
Tabel users
Semua pengguna sistem
Cara menjelaskan
"Tabel users menyimpan semua pengguna sistem. Ada tiga role: admin, kasir, dan customer. Kolom role bertipe ENUM — artinya MySQL hanya menerima tiga nilai itu, tidak bisa diisi sembarangan. Kolom email diberi constraint UNIQUE agar tidak ada dua user dengan email yang sama."
Tabel tables (meja)
Data fisik meja restoran
Cara menjelaskan
"Tabel tables menyimpan data meja restoran — nomor meja, kapasitas, dan statusnya. Status berupa ENUM: tersedia, terisi, atau dipesan. Status ini berubah otomatis lewat trigger dan stored procedure, sehingga kondisi meja selalu akurat tanpa update manual."
Tabel categories + menus
Relasi 1:N — 4 kategori, 9 menu
Cara menjelaskan
"Tabel categories menyimpan 4 kategori: Makanan Utama, Minuman, Snack, dan Dessert. Tabel menus terhubung ke categories lewat Foreign Key category_id. Relasinya 1:N — satu kategori bisa punya banyak menu, tapi setiap menu hanya masuk ke satu kategori. FK ini dikonfigurasi ON DELETE RESTRICT, artinya kategori tidak bisa dihapus kalau masih punya menu."
Data yang sudah ada
Makanan Utama: Nasi Goreng, Mie Ayam, Soto Ayam, Ayam Bakar. Minuman: Es Teh, Jus Alpukat, Kopi Hitam. Dessert: Es Krim. Snack: Kentang Goreng.
Tabel bookings
Reservasi meja oleh customer
Cara menjelaskan
"Tabel bookings menyimpan data reservasi meja. Tabel ini punya dua FK: ke users untuk mencatat siapa yang booking, dan ke tables untuk mencatat meja mana yang dipesan. Ada kolom jam_mulai dan jam_selesai bertipe TIME untuk pengecekan overlap jadwal di stored procedure."
Tabel orders + order_items
Relasi M:N — yang paling penting!
Cara menjelaskan — fokus di sini!
"Ini bagian terpenting dari ERD kami. Tabel orders menyimpan header transaksi, sedangkan order_items menyimpan detail item-itemnya. Hubungan antara orders dan menus adalah Many-to-Many — satu order bisa berisi banyak menu, dan satu menu bisa muncul di banyak order yang berbeda. Tabel order_items adalah tabel pivot yang menjembatani relasi M:N ini, sekaligus menyimpan data tambahan seperti qty, harga satuan, dan subtotal."
Ini PASTI ditanya dosen. Pastikan semua anggota kelompok bisa jawab kenapa ini M:N bukan 1:N.
Tabel payments
Relasi 1:1 dengan orders
Cara menjelaskan
"Tabel payments menyimpan data pembayaran. Relasinya 1:1 dengan orders — satu order hanya boleh punya satu pembayaran, dijamin lewat constraint UNIQUE pada kolom order_id. Kolom bukti_transfer menyimpan path foto bukti transfer, yang akan diupload lewat fitur upload di Laravel."
Menjelaskan index
Cara membuka penjelasan index
Mulai dengan analogi daftar isi buku
Pembuka yang bagus
"Sebelum saya jelaskan index-nya, analoginya begini: bayangkan tabel bookings punya 100.000 baris. Kalau kita cari booking tanggal 10 Agustus tanpa index, MySQL harus baca dari baris pertama sampai terakhir. Dengan index, dia langsung lompat ke tanggal yang dicari — seperti daftar isi buku."
Jelaskan 5 index
"Kami membuat 5 index dengan dua jenis: 3 single-column dan 2 composite index. Pertama, idx_bookings_tanggal — dipilih karena setiap ada permintaan booking, sistem selalu filter berdasarkan tanggal untuk cek jadwal."
Lanjutkan
"Kedua, idx_orders_status — kasir dan dapur selalu filter WHERE status='diproses'. Ketiga, idx_menus_category_status — ini composite index dua kolom sekaligus, karena query menu SELALU filter kategori DAN status tersedia bersamaan. Composite index lebih efisien dari dua index terpisah karena satu kali baca index sudah cukup."
Penutup index
"Keempat, idx_order_items_order — tabel order_items adalah tabel dengan baris terbanyak karena satu order bisa punya banyak item. Kelima, idx_bookings_table_tanggal — composite index untuk cek overlap jadwal di stored procedure, filter meja sekaligus tanggal."

Menjelaskan stored procedure
Pengantar stored procedure
Kenapa pakai SP?
Ucapkan ini
"Stored procedure adalah kumpulan logika bisnis yang disimpan di database. Keuntungannya: daripada menulis validasi yang sama berulang kali di aplikasi, cukup tulis sekali di SP dan panggil dengan CALL dari mana saja. Kami membuat 3 stored procedure dengan operasi berbeda-beda, semuanya menggunakan parameter IN dan OUT serta error handling dengan ROLLBACK."
sp_buat_booking — paling kompleks
Demo live ini!
Cara menjelaskan
"SP pertama, sp_buat_booking, memiliki dua lapis validasi sebelum menyimpan data. Pertama, cek apakah jumlah tamu tidak melebihi kapasitas meja. Kedua, cek apakah ada booking lain di meja yang sama pada tanggal dan jam yang tumpang tindih — overlap check. Kalau kedua validasi lolos, data disimpan dalam satu TRANSACTION: INSERT ke bookings dan UPDATE status meja, keduanya harus berhasil bersama-sama."
CALL sp_buat_booking(3, 1, '2026-08-10', '18:00', '20:00', 2, 'Anniversary', @res); SELECT @res;
Tunjukkan hasilnya: @res akan berisi "SUKSES: Booking berhasil dibuat dengan ID X". Coba sekali lagi dengan jam yang sama → "GAGAL: Meja sudah dipesan".
sp_buat_order + sp_proses_pembayaran
SP kedua dan ketiga
sp_buat_order
"SP kedua, sp_buat_order, menerima daftar menu dalam format JSON. SP ini loop setiap item, cek statusnya tersedia atau tidak, ambil harganya, lalu insert ke order_items. Total harga dihitung otomatis dari akumulasi seluruh item, kemudian diupdate ke tabel orders."
sp_proses_pembayaran
"SP ketiga, sp_proses_pembayaran, menangani pembayaran dengan dua validasi: cek apakah order belum pernah dibayar, dan cek apakah jumlah yang dibayar mencukupi total order. Kalau valid, satu transaksi mengeksekusi tiga hal sekaligus: insert ke payments, update status order jadi selesai, dan update status meja kembali jadi tersedia."
Menjelaskan trigger
Pengantar trigger
Sensor otomatis database
Ucapkan ini
"Trigger adalah prosedur yang jalan otomatis saat ada kejadian di tabel — INSERT, UPDATE, atau DELETE — tanpa perlu dipanggil manual. Analoginya seperti sensor pintu otomatis: begitu ada yang lewat, pintu terbuka sendiri. Kami membuat 4 trigger dengan 4 jenis berbeda."
4 trigger — variasi BEFORE dan AFTER
Jelaskan satu per satu
Trigger 1 — BEFORE INSERT
"Trigger pertama, trg_before_insert_order_items, jenis BEFORE INSERT. Karena jalan sebelum data disimpan, trigger ini bisa mengubah nilai yang akan masuk. Fungsinya dua: validasi apakah menu yang dipesan statusnya tersedia — kalau habis langsung lempar error — dan otomatis menghitung subtotal dari qty dikali harga satuan."
Trigger 2 — AFTER INSERT
"Trigger kedua, trg_after_insert_order_items, jenis AFTER INSERT. Setelah item baru masuk ke order_items, trigger ini otomatis mengupdate kolom total_harga di tabel orders agar nilainya selalu akurat tanpa perlu dihitung ulang dari aplikasi."
Trigger 3 — AFTER UPDATE
"Trigger ketiga, trg_after_update_bookings, jenis AFTER UPDATE pada tabel bookings. Trigger ini memantau perubahan status booking. Kalau status berubah jadi dibatalkan, meja otomatis dikembalikan ke tersedia. Kami menggunakan OLD.status dan NEW.status untuk membandingkan nilai sebelum dan sesudah perubahan."
Trigger 4 — BEFORE UPDATE
"Trigger keempat, trg_before_update_orders, jenis BEFORE UPDATE. Fungsinya sebagai penjaga integritas status — mencegah perubahan status yang tidak logis, seperti order yang sudah selesai diubah kembali ke pending, atau order yang sudah dibatalkan diaktifkan lagi."
Menjelaskan user privilege
Pengantar + 3 user
Prinsip least privilege — wajib disebut!
Pengantar
"User privilege mengatur siapa boleh melakukan apa di database. Kami menerapkan prinsip least privilege — setiap user hanya diberi akses yang benar-benar dibutuhkan untuk tugasnya, tidak lebih."
admin_restoran
"User pertama, admin_restoran — mendapat ALL PRIVILEGES karena bertanggung jawab atas seluruh sistem termasuk manajemen user dan struktur database."
kasir_restoran — update terbaru
"User kedua, kasir_restoran — hanya SELECT, INSERT, UPDATE pada tabel yang relevan. Kasir tidak bisa DELETE, tidak bisa akses laporan keuangan, dan untuk tabel users hanya bisa lihat kolom id dan name saja — tidak bisa lihat email atau password."
customer_app
"User ketiga, customer_app — akses paling terbatas. Hanya bisa SELECT menu dan categories, serta INSERT booking dan order. Tidak bisa akses tabel users atau payments sama sekali."
Sebutkan kata "least privilege" — istilah teknis DCL yang dikenal dosen.

Antisipasi pertanyaan dosen — 10 pertanyaan
"Kenapa pakai 8 tabel? Bukankah minimal 5?"
Jawab: "Kami merancang 8 tabel karena mengikuti kebutuhan bisnis nyata sistem restoran. Kalau hanya 5, kita tidak bisa pisahkan categories dari menus, tidak ada tabel pivot order_items untuk relasi M:N, dan tidak ada tabel payments yang terpisah. Setiap tabel punya peran yang tidak bisa digabung tanpa melanggar normalisasi 3NF."
"Jelaskan normalisasi 3NF pada database ini?"
Jawab: "Database kami memenuhi 3NF. Pertama, semua kolom bernilai atomik. Kedua, semua kolom non-key bergantung penuh pada Primary Key — memenuhi 2NF. Ketiga, tidak ada dependensi transitif. Contoh nyata: nama kategori tidak kita simpan di tabel menus, melainkan di tabel categories yang direferensikan lewat FK category_id."
"Kenapa order_items itu relasi M:N, bukan 1:N saja?"
Jawab: "Karena satu order bisa berisi banyak menu yang berbeda, dan satu menu yang sama bisa muncul di banyak order berbeda. Contoh: Nasi Goreng bisa dipesan di Order #1 oleh Andi, dan Order #5 oleh Sari. Tanpa tabel pivot order_items, kita tidak bisa simpan hubungan ini tanpa redundansi yang melanggar normalisasi."
"Apa bedanya BEFORE dan AFTER trigger?"
Jawab: "BEFORE trigger jalan sebelum data disimpan — kita masih bisa mengubah nilai yang akan masuk menggunakan SET NEW.kolom. Kami pakai ini untuk menghitung subtotal dan validasi status menu. AFTER trigger jalan setelah data sudah tersimpan — tidak bisa ubah data yang baru masuk, tapi bisa update tabel lain. Kami pakai ini untuk update total_harga di orders dan sinkronisasi status meja."
"Apa itu TRANSACTION dan kenapa dipakai di SP?"
Jawab: "TRANSACTION memastikan sekumpulan operasi SQL berhasil semua atau gagal semua. Contoh di sp_buat_booking: kita INSERT ke bookings dan UPDATE status meja dalam satu transaksi. Kalau INSERT berhasil tapi UPDATE gagal, ROLLBACK memastikan INSERT tadi dibatalkan juga. Tidak mungkin ada booking tapi meja masih berstatus tersedia."
"Kenapa pakai DELIMITER $$ di stored procedure?"
Jawab: "Karena di dalam body stored procedure ada banyak titik koma untuk memisahkan setiap statement. Kalau pemisah default tetap titik koma, MySQL kira prosedur selesai di titik koma pertama. Dengan DELIMITER $$, MySQL membaca seluruh blok BEGIN...END sebagai satu unit. Setelah selesai, DELIMITER dikembalikan ke titik koma."
"Kenapa tidak membuat index di semua kolom saja?"
Jawab: "Index mempercepat SELECT tapi memperlambat INSERT, UPDATE, dan DELETE — karena setiap perubahan data, index juga harus diperbarui. Index juga memakan storage tambahan. Jadi kita hanya buat index pada kolom yang benar-benar sering jadi filter pencarian."
"Kenapa kasir tidak bisa lihat semua kolom tabel users?"
Jawab: "Ini penerapan prinsip least privilege. Kasir tidak perlu tahu email dan password hash pelanggan lain — itu data sensitif. Kasir hanya butuh tahu nama customer untuk konfirmasi order, jadi kami batasi akses kasir ke kolom id dan name saja menggunakan GRANT SELECT (id, name) ON users."
"Apa itu composite index dan kapan dipakai?"
Jawab: "Composite index adalah index yang dibuat pada dua kolom atau lebih sekaligus. Kami pakai untuk menus(category_id, status) karena query menu SELALU filter keduanya bersamaan: WHERE category_id=1 AND status='tersedia'. Composite index lebih efisien dari dua index terpisah karena MySQL cukup satu kali baca index."
"Apa perbedaan ON DELETE RESTRICT, CASCADE, dan SET NULL?"
Jawab: "RESTRICT menolak penghapusan jika masih ada data terkait — kami pakai di categories → menus agar kategori tidak bisa dihapus kalau masih ada menu. CASCADE menghapus data anak secara otomatis — kami pakai di orders → order_items agar item ikut terhapus kalau order dihapus. SET NULL mengosongkan FK anak — kami pakai di bookings → orders agar kalau booking dihapus, order tetap ada tapi booking_id-nya jadi NULL."
